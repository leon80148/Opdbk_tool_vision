## 1. 專案背景與目標

### 1.1 專案背景

診所目前使用既有 HIS/掛號系統（Windows + DBF），櫃檯人員於病患掛號時需同時處理：

* 病歷號輸入、掛號、收費
* 協助醫師安排檢驗、檢查與衛教
* 確認病患資料完整度（聯絡資料等）

**現況問題：**

* 衛教、檢查追蹤、疫苗推廣，多仰賴櫃檯或醫師「記憶」或「臨時查詢」，容易遺漏。
* 檢驗數據分散在 `CO18H` 等資料表中，結構屬於「垂直表」，不利於快速閱讀。
* 要判斷病患是否有逾期檢查 / 適合衛教，需人工翻找檢查紀錄與檢驗值，耗時且易出錯。

本專案希望在不修改既有 HIS 的前提下，以 **Electron 桌面輔助工具** 的形式，
透過 DBF/ODBC 讀取 HIS 資料，在櫃檯「輸入病歷號的瞬間」顯示整合資訊的決策面板。

### 1.2 v3.0 更新重點

* **移除** 對 `lab_base` 寬表的依賴。
* **全面改採** `CO18H` 垂直表作為檢驗數據的唯一來源。
* **新增** 本機 SQLite 中介層：

  * 將 `CO18H` 中關鍵檢驗項目轉置為「寬表」格式快取。
  * 前端所有檢驗矩陣查詢一律走 SQLite，確保秒級回應。
* **新增** 全域熱鍵啟動功能，櫃檯可一鍵喚出助手。
* **新增** INI 設定檔機制，可調整：

  * DBF 資料庫根目錄
  * SQLite 檔案路徑
  * 全域熱鍵組合
  * 檢驗同步頻率（分鐘）

> 備註：原先規劃的「欠費提醒功能」已於 v3.0 中 **移除**。

### 1.3 產品目標

1. 在櫃檯掛號或接聽電話時，提供病患的關鍵資訊一覽表，包括：

   * 慢性病風險與重要檢驗數值（HBA1C、CHOL、eGFR…）
   * 逾期或即將逾期的檢查追蹤（超音波、內視鏡等）
   * 待安排的衛教
   * 疫苗施打狀態
   * 基本資料是否完整（電話、地址）

2. 協助診所達成：

   * 提升預防保健與衛教覆蓋率。
   * 系統化推廣疫苗與自費檢查。
   * 提高檢查追蹤與回診率，減少逾期未追蹤的病患。

## 3. 使用者故事 (User Stories)

1. **衛教與疫苗推廣**

   > 作為櫃檯人員，我想要一眼看出病患是否有「待執行的衛教」或「符合施打資格的疫苗（如帶狀皰疹、肺炎鏈球菌）」 ，以便主動推廣或安排。

2. **慢性病風險與檢驗單列印**

   > 作為櫃檯人員，我想要知道病患的「慢性病風險指標（糖尿病 / 高血壓 / 高血脂）」 ，以便在醫師看診前先印出相關檢驗單。

3. **檢查追蹤提醒**

   > 作為櫃檯人員，我想要知道病患上次做「超音波」或「內視鏡」的時間，若已逾期，可提醒病患安排檢查。

4. **快速喚出工具**

   > 作為櫃檯人員，我希望可以用鍵盤熱鍵，從任何畫面快速叫出診所櫃檯智慧助手，節省滑鼠點來點去的時間。

#### 4.1.3 標籤 (Tags)

* VIP

> 註：Tag 內容可來自 HIS 既有欄位或外部設定，v3.0 可先以讀取某欄位文字方式實作，後續再細緻化。

#### 4.1.4 快速警示燈號

以圖示 + 顏色表示關鍵狀態：

1. **衛教提醒燈**

   * 顏色：

     * 🔴：異常且明確建議安排衛教
     * 🟠：偏高或需注意，可視情況安排衛教
     * 🟢：數值在正常範圍，不需特別提醒
   * 基本規則（可於設定檔中調整）：

     * 例：`HBA1C > 7.0` → 🟠 或 🔴

2. **資料完整度燈（可選）**

   * 依病患基本資料欄位（手機 `mtelh`、地址 `maddr`）是否為空判斷：

     * 🟠：部分資料缺漏
     * 🔴：重要聯絡資料皆缺漏
     * 🟢：資料完整
   * 詳細內容在 Smart Action List 顯示。

> 欠費燈號已在 v3.0 需求中移除，不再顯示或計算欠費狀態。

### 4.2 待辦事項與警示 (Smart Action List)

系統自動產生櫃檯 Check List，依優先級排列。

#### 4.2.1 顯示形式

* 清單式（List）呈現，每一項包含：

  * Icon/顏色（代表緊急程度）
  * 標題（簡短描述）
  * 說明文字（必要時）
  * 相關時間或數值（例如上次檢查日期）

#### 4.2.2 現階段支援規則

1. **衛教提醒**

* 條件範例：

  * `HBA1C > 7.0`（從 SQLite 寬表中取得）
* 顯示文案範例：

  * 🟠 需安排糖尿病衛教（HBA1C：8.2，檢驗日：2025/03/01）

2. **檢查追蹤**

* 條件：

  * 最近一次指定檢查（如「腹部超音波」、「胃鏡」、「大腸鏡」）距今日超過建議週期。
* 資料來源：

  * `CO02M` / `CO03M` 中的檢查醫令 / labno / dno。
* 顯示文案範例：

  * 🔵 腹部超音波已逾期（上次：2023/01/15，建議週期：1 年）

3. **資料補填**

* 條件：

  * `mtelh`（手機）、`maddr`（地址）其中任一為空。
* 顯示文案範例：

  * 🟡 聯絡資料不完整：請補填手機 / 地址

> 規則邏輯建議集中於設定檔 / 規則模組，便於未來新增/調整。

### 4.6 設定與 INI 檔管理 (Configuration via INI File)

#### 4.6.1 需求說明

* 系統需透過 `config.ini` 讓診所可簡單調整以下設定：

  * DBF 資料庫根目錄 (`dbf_root`)
  * SQLite 檔案路徑 (`sqlite_path`)
  * 全域熱鍵 (`global`)
  * LabCodeMap JSON 檔案位置 (`lab_code_map`)
  * 檢驗同步頻率 (`interval_minutes`)

#### 4.6.2 設定檔位置與命名

* 檔名：`config.ini`
* 預設位置：與主程式執行檔同層，例：`C:\ClinicAssistant\config.ini`
* 若缺少 `config.ini`：

  * 使用程式內建預設值
  * 可於 Log 記錄提示

#### 4.6.3 設定檔格式範例

```ini
[database]
dbf_root = D:\HIS\DBF\
sqlite_path = C:\ClinicAssistant\data\lab_cache.db

[hotkey]
global = Ctrl+Alt+C

[labs]
lab_code_map = C:\ClinicAssistant\config\lab_codes.json

[sync]
; 同步間隔（分鐘）
; 建議範圍 5~60 分鐘，0 或負數視為停用週期性同步
interval_minutes = 10
```

#### 4.6.4 行為規則

* 啟動時由 Main Process 讀取 `config.ini`，欄位缺失則採用預設值。
* `interval_minutes`：

  * > 0：每 N 分鐘背景增量同步 CO18H → SQLite
  * =0 或無設定：僅於啟動時同步一次，不啟用週期性同步
* 熱鍵註冊失敗時需提示使用者。

## 5. 資料庫映射與 CO18H → SQLite 同步流程 (Schema Mapping & Sync Flow)

### 5.1 主要資料來源表（HIS / DBF）

* `CO18H`：檢驗資料（垂直表）

  * 重要欄位（實際依 HIS 而定）：

    * `kcstmr`：病歷號
    * `hitem`：檢驗項目代碼
    * `hdate`：檢驗日期
    * `hval`：檢驗結果值
    * `hunit`：單位

* `CO02M`：醫令 / 處方（檢查與疫苗）

  * `dno`：藥品 / 檢查代碼
  * `date`：執行日期

* `CO03M`：就診 / 檢查相關紀錄（不再用於欠費，只用於檢查歷程等）

* `VISHFAM.DBF`：病患基本資料

  * `kcstmr`：病歷號
  * `mname`：姓名
  * `pid`：身分證號
  * `sex`：性別
  * `age` 或出生年月日
  * `mtelh`：手機
  * `maddr`：地址

> 編碼需處理 Big5 vs UTF-8 差異，避免亂碼。

---

### 5.2 SQLite 資料表設計（建議）

> 以下為邏輯設計，實作時可微調欄位名稱。

#### 5.2.1 `lab_results_raw`（原始檢驗快取）

* 用途：

  * 儲存從 `CO18H` 抓下來的檢驗紀錄，做為快取與追溯來源。
* 欄位示例：

  * `id` (PK)
  * `kcstmr`：病歷號
  * `hitem`：檢驗項目代碼
  * `hdate`：檢驗日期
  * `hval`：數值（文字）
  * `hunit`：單位
  * `source_updated_at`：來源資料最後更新時間（可選）
  * `created_at` / `updated_at`

#### 5.2.2 `lab_results_wide`（寬表，供前端查詢）

* 用途：

  * 提供前端一次取得各類檢驗的最新值，避免每次 pivot。
* 欄位示例：

  * `kcstmr`：病歷號 (PK)
  * `last_dm_date`：DM 區最後更新日期（可選）
  * `hba1c` / `hba1c_date`
  * `uacr` / `uacr_date`
  * `egfr` / `egfr_date`
  * `chol` / `chol_date`
  * `ldl_c` / `ldl_c_date`
  * `tg` / `tg_date`
  * `bmi` / `bmi_date`
  * `anti_hcv` / `anti_hcv_date`
  * `hbsag` / `hbsag_date`
  * `updated_at`

#### 5.2.3 `sync_meta`（同步狀態紀錄）

* 用途：

  * 記錄 CO18H → SQLite 同步的進度與狀態。
* 欄位示例：

  * `id` (固定 1)
  * `last_hdate_synced`：上次同步完成的最大 `hdate`
  * `last_sync_started_at`
  * `last_sync_finished_at`
  * `last_sync_status`：成功 / 失敗 / 局部成功
  * `last_sync_error`：錯誤訊息（可選）

### 5.3 初始匯入流程（第一次部署 / 空資料庫啟動）

**目的：** 在第一次使用時，建立完整基礎快取與寬表。

#### 5.3.1 流程步驟

1. **檢查 SQLite 狀態**

   * 若 `lab_results_raw` 或 `sync_meta` 不存在 → 視為第一次匯入。

2. **讀取設定**

   * 從 `config.ini` 取得：

     * `dbf_root`：CO18H 所在目錄
     * `labs.lab_code_map`：需要關注的 `hitem` 列表

3. **從 CO18H 匯入指定範圍資料**

   * 建議只匯入「最近 N 年」的檢驗（例如 2–3 年），避免資料量過大。
   * 範例條件：

     * `hdate >= (今天 - 3 年)`
     * `hitem IN (LabCodeMap 中所有代碼)`

4. **寫入 `lab_results_raw`**

   * 將上述資料批次寫入 SQLite。

5. **建立 / 更新 `lab_results_wide`**

   * 依 `kcstmr + hitem` 分組，取 `MAX(hdate)` 的紀錄作為最新值。
   * 依 `LabCodeMap` 將結果轉成寬表欄位，更新 `lab_results_wide`。

6. **更新 `sync_meta`**

   * `last_hdate_synced = 此次匯入資料中的最大 hdate`
   * `last_sync_started_at / last_sync_finished_at / last_sync_status = success`

#### 5.3.2 使用者體驗

* 初始匯入可能花費數十秒～數分鐘（依資料量而定）。
* 建議在首次啟動時顯示「初始化中」訊息。
* 完成後即進入正常使用狀態。

---

### 5.4 增量同步流程（Incremental Sync）

**目的：** 定期將 CO18H 中新增或更新的檢驗紀錄同步至 SQLite。

#### 5.4.1 同步邏輯

1. **讀取 `sync_meta`**

   * 取得 `last_hdate_synced`，若為空，則回到初始匯入流程。

2. **從 CO18H 擷取新資料**

   * 查詢條件：

     * `hdate > last_hdate_synced`
     * `hitem IN (LabCodeMap)`

3. **寫入 / 更新 `lab_results_raw`**

   * 對於新抓到的每筆紀錄：

     * 若主鍵設計為 (`kcstmr`, `hitem`, `hdate`)，則：

       * 若無此組合 → INSERT
       * 若已存在（例如 HIS 修正值）→ UPDATE

4. **更新 `lab_results_wide`**

   * 針對受影響的病患 (`kcstmr`) 重新計算：

     * 對該病患在 `lab_results_raw` 的資料依 `hitem` 分組，取最新 `hdate`。
     * 更新該病患在 `lab_results_wide` 的欄位。

5. **更新 `sync_meta`**

   * `last_hdate_synced = 本次抓取資料中的最大 hdate`
   * 更新同步時間與狀態、錯誤訊息（如有）。

#### 5.4.2 同步觸發時機

* **啟動時**：

  * App 啟動後背景執行一次增量同步。

* **週期性背景同步**（由 `config.ini` 控制）：

  * 依 `sync.interval_minutes` 設定（例如 10 分鐘）啟動定時任務。
  * `interval_minutes = 0` 或缺值 → 不啟用週期性同步，仅啟動時同步一次。

* **手動同步（可選功能）**：

  * 提供隱藏的「立即同步」按鈕給維護人員或工程師使用（非 v3.0 必要，但可預留）。

---

### 5.5 前端查詢流程（櫃檯操作視角）

1. 櫃檯輸入病歷號或從熱鍵喚出病人查詢畫面。
2. 前端透過 IPC / REST 呼叫 Node.js：

   * 查詢目標包含：

     * 病患基本資料（來自 DBF 或 SQLite 快取）
     * 檢驗矩陣（來自 `lab_results_wide`）
     * 檢查 / 預約歷程（來自 CO02M / CO03M）
3. Node.js 只查 **SQLite 中的檢驗寬表**，不直接掃 CO18H。
4. 畫面在 1 秒內呈現關鍵資訊。

> 即使背景同步尚未完成，櫃檯仍可使用最近一次同步的資料，不會卡住。

---

### 5.6 錯誤處理與回復策略

* 若與 CO18H 連線失敗：

  * 同步 job 記錄錯誤於 Log，`sync_meta.last_sync_status = failed`。
  * 前端可顯示「檢驗同步異常，顯示為上次同步資料」。

* 若 SQLite 損毀或無法開啟：

  * 可自動重新建立 SQLite 檔，重新跑初始匯入。
  * 或請維護人員刪除舊檔後重新啟動程式。

## 7. 非功能性需求 (NFR)

### 7.1 效能

* 病歷號查詢 → 主要畫面（Header + Smart Action List + 檢驗矩陣）在 **1 秒內**完成。
* 右側完整矩陣與歷程資料在 **3 秒內**完成載入。
* App 冷啟動時間建議 **10 秒以內**（含 SQLite 初始化）。

### 7.2 穩定性

* 需支援每日 80–150 人、5 台電腦同時使用，不影響 HIS 正常運作。
* 背景同步應不會長時間鎖住 CO18H / DBF 檔案。

### 7.3 資料同步

* 預設每 10 分鐘從 CO18H 增量同步到 SQLite（interval 可由 `config.ini` 調整）。
* 失敗時不影響畫面查詢，但需在 Log 中可追踪。

### 7.4 安全性與隱私

* 僅部署於院內網路，不提供外部連線。
* Log 不紀錄完整身分證號，可採遮蔽處理。
* SQLite 檔可考慮權限限制（僅允許指定帳號存取）。

### 7.5 相容性與安裝

* 作業系統：Windows 10 以上。
* 安裝方式：提供標準安裝程式，安裝後產生預設 `config.ini`。
* 需處理 Big5 / UTF-8 編碼差異，確保病患姓名等欄位不亂碼。

---

## 8. 限制條件與假設

* 以單一診所為設計前提，多院所版本另行規劃。
* 不更動現有 HIS / DBF Schema，只以唯讀方式存取。
* 檢驗代碼（`hitem`）、檢查代碼（`dno`）需由 HIS 廠商提供並維護。
* 若病患基本資料在 HIS 本身未維護完整，本系統只能提示補填，無法自動補資料。

## 10. 成功指標與衡量方式 (KPI)

1. **檢查追蹤成效**

   * 指標：

     * 超過建議檢查週期仍未追蹤的病患比例（導入前 vs 導入後）。
   * 目標方向：

     * 導入後 6–12 個月內，逾期未追蹤比例顯著下降（具體 Y% 可與診所共同設定）。

2. **櫃檯主觀滿意度**

   * 評估方式：

     * 以問卷 / 訪談調查櫃檯人員，評分範圍 0–100 分。
   * 目標：

     * 平均滿意度分數 **≥ 80 分**。
   * 參考問題：

     * 是否覺得有幫助記得提醒病患做檢查 / 衛教？
     * 是否覺得畫面速度足夠？
     * 熱鍵喚出是否方便？
